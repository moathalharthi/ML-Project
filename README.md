# ML-Project
# Credit Default Prediction 

A machine learning project that predicts whether a credit card customer will default on their next monthly payment, built on the Taiwan Default of Credit Card Clients dataset using PyCaret 3.x.

---


## Problem

Banks lose money when customers default on credit-card debt. Catching likely defaulters early lets the bank intervene — restructure payments, lower limits, or close exposure — *before* the loss is realised.

This project trains a binary classifier that, given a customer's recent payment history and demographics, outputs the **probability that they will default next month**.

## Methodology

The notebook follows a 13-section CRISP-DM-style arc:

1. **ART analysis** — Availability, Representativeness, Trust
2. **Data quantity & quality** — completeness, duplicates, dtype audit
3. **Feature engineering** — six behavioural aggregates (utilization, payment ratio, delay metrics)
4. **Predictive power** — Pearson correlation + mutual information ranking
5. **PyCaret setup** — stratified 5-fold CV, SMOTE inside folds, declared categoricals
6. **Compare all models** — full leaderboard ranked by AUC
7. **LightGBM vs Logistic Regression** — head-to-head with side-by-side metrics
8. **Hyperparameter tuning** — Optuna, 50 trials, optimised for AUC
9. **Threshold optimization** — sweep 0.10–0.90, pick F1-optimum
10. **Predictions & risk segmentation** — bucket customers as High / Medium / Low risk
11. **Success criteria evaluation** — vs the four required thresholds
12. **Deployment** — pickle, FastAPI, Docker, Gradio
13. **Discussion & reflection** — limitations, fairness, next steps

## Dataset

**Default of Credit Card Clients** (Yeh & Lien, 2009)

- **Source**: [UCI ML Repository](https://archive.ics.uci.edu/dataset/350/default+of+credit+card+clients)
- **Distributed via**: PyCaret's `get_data('credit')` (24,000-row sample of the original 30,000)
- **Period**: April–September 2005
- **Region**: Taiwan
- **Target**: `default` (1 = default next month, 0 = no default)
- **Class balance**: 78% no-default / 22% default

> ⚠️ **Distributional caveat**: The data is Taiwan, 2005. Deploying in another country or decade requires retraining, not just reloading the pickle.

##  Data Dictionary

| Column Name | Description | Values / Interpretation |
| :--- | :--- | :--- |
| **ID** | ID of each client | Unique Identifier |
| **LIMIT_BAL** | Amount of given credit (NT dollar) | Includes individual & family credit |
| **SEX** | Gender | 1 = Male, 2 = Female |
| **EDUCATION** | Education level | 1=grad school, 2=university, 3=high school, 4=others |
| **MARRIAGE** | Marital status | 1=married, 2=single, 3=others |
| **AGE** | Age in years | Numerical |
| **PAY_1 - 6** | Repayment status (Sept - April) | -1=pay duly, 1=delay 1 month, 2=delay 2 months... |
| **BILL_AMT1 - 6** | Amount of bill statement (Sept - April) | Amount in NT dollar |
| **PAY_AMT1 - 6** | Amount of previous payment (Sept - April) | Amount in NT dollar |
| **default** | **Target Variable** | 1 = Yes (Default), 0 = No |

## Limitations

- 22% prevalence caps achievable F1 on a balanced metric
- No cost matrix specified — if a missed default actually costs N× a false alarm, cost-sensitive learning would beat threshold optimization
- No fairness audit performed (TPR/FPR parity by `SEX`, `EDUCATION`, `MARRIAGE`) — required before deployment in regulated settings
- Predicted probabilities are not yet calibrated (Platt or isotonic) — fine for ranking, careful when reporting "70% probability of default" verbatim
